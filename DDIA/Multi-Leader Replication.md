При использовании [[Leader-based Replication]] возникает проблема: при недоступности мастера все операции записи в базу данных блокируются. Расширением данной модели является модель Multi-Leader Replication, в которой записи могут поступать на любую ноду, которая затем передаёт изменения остальным нодам кластера. В такой конфигурации каждый лидер одновременно является репликой для других лидеров.

Сценарии, в которых применима репликация с несколькими лидерами:
- Развёртывание в нескольких датацентрах. В каждом датацентре создаётся лидер, который реплицирует записи в лидеры других датацентров.
- Клиенты, которые должны уметь работать без доступа к интернету. Например, приложение на телефон с календарём, в котором созданные события реплицируются на другие устройства пользователя. **CouchDB** спроектирован для такого режима работы.
- Приложения, поддерживающие совместное редактирование документов.

### Обработка конфликтов записи
При поступлении записей, которые изменяют одинаковые данные, в разные лидеры, может возникнуть конфликт записи. Верхнеуровнево, есть несколько подходов к их разрешению:
- Синхронно реплицировать запись всем другим лидерам, перед тем как подтверждать её. При этом теряется основное преимущество репликации с несколькими лидерами, так как невозможно обрабатывать операции записи независимо каждой репликой.
- Избегать конфликтов через направление записей, изменяющих одни и те же данных, в один и тот же датацентр.
- Приведение к согласованному состоянию. После того, как все изменения будут реплицированы, все реплики должны сойтись на одинаковых значениях в конфликтующих записях. Существует несколько подходов:
    1. Last Write Wins (LWW). Каждой записи присваивается отметка времени (в общем случае уникальный ID), финальной считается запись с максимальной отметкой (максимальным ID). Возможна потеря данных.
    2. Каждой реплике присваивается уникальный ID, записи в реплику с большим ID имеют приоритет перед записями в реплику с меньшим ID. Возможна потеря данных.
    3. Провести слияние конфликтующих значений.
    4. Сохранить все записи и реализовать логику разрешения конфликта на стороне приложения.
- Собственная логика разрешения конфликтов на записи или на чтении.
- Автоматическое разрешение конфликтов:
    1. Conflict-free replicated datatypes. Структуры данных, которые позволяют параллельное редактирование несколькими пользователями и автоматически разрешают конфликты.
    2. Mergable persistent data structures. Сохраняют историю изменений и используют функции слияния.
    3. Operational transformation. Параллельное редактирование упорядоченного списка записей (например, список символов в текстовом документе).
